## 1. Maven依赖问题

### 1.1 Spring AI BOM版本找不到

**错误信息：**
```
org.springframework.ai:spring-ai-bom:pom:1.0.0-M4 was not found in https://maven.aliyun.com/repository/public
```

**原因：**
- Spring AI 1.0.0-M4是里程碑版本，阿里云镜像未同步
- 阿里云镜像主要同步主流包，Spring AI等较新包可能没有

**解决方案：**
1. 使用较新的里程碑版本 `1.0.0-M6`
2. 添加Spring官方仓库

```xml
<properties>
    <spring-ai.version>1.0.0-M6</spring-ai.version>
</properties>

<repositories>
    <repository>
        <id>spring-milestones</id>
        <url>https://repo.spring.io/milestone</url>
        <snapshots>
            <enabled>true</enabled>
        </snapshots>
    </repository>
</repositories>
```

---

### 1.2 依赖版本缺失

**错误信息：**
```
'dependencies.dependency.version' for org.springframework.ai:spring-ai-pgvector-store-spring-boot-starter:jar is missing
```

**原因：**
- Spring Boot版本与Spring AI版本不兼容
- 版本对应关系：
  - Spring AI 1.0.0-M4 → Spring Boot 3.2.x
  - Spring AI 1.0.0 (正式版) → Spring Boot 3.3.x

**解决方案：**
- 使用 Spring AI 1.0.0-M6 + Spring Boot 3.2.5

## 3. AI接口配置问题

### 3.1 URL路径重复

**错误信息：**
```
java.io.FileNotFoundException: https://dashscope.aliyuncs.com/compatible-mode/v1/v1/chat/completions
```

**原因：**
- `base-url` 配置了 `https://dashscope.aliyuncs.com/compatible-mode/v1`
- Spring AI会自动追加 `/v1/chat/completions`
- 导致路径变成 `/v1/v1/chat/completions`

**解决方案：**
```yaml
spring.ai:
  openai:
    base-url: https://dashscope.aliyuncs.com/compatible-mode  # 不要包含/v1
```

## 4. 流式输出问题

### 4.1 中文乱码

**现象：**
```
data:娓data:娓呮€濊矾銆佸涔銆佸涔�
data:鏂扮煡璇嗭紝鎴栬€�
```

**原因：**
- 流式响应（SSE）未强制指定UTF-8编码
- 默认编码可能不是UTF-8

**解决方案：**
添加编码过滤器：

```java
@Configuration
public class WebConfig {
    
    @Bean
    public CharacterEncodingFilter characterEncodingFilter() {
        CharacterEncodingFilter filter = new CharacterEncodingFilter();
        filter.setEncoding("UTF-8");
        filter.setForceEncoding(true);
        return filter;
    }
}
```

---

### 4.2 SSE格式的data:前缀

**现象：**
```
data:我都很乐意！
data:✨  
data:有什么我可以帮你的
```

**原因：**
这是SSE（Server-Sent Events）协议的标准格式，每条消息必须以 `data:` 开头。

**说明：**
- `data:` 是协议格式，不是bug
- 前端负责解析并去掉 `data:` 前缀

**前端解析示例：**
```javascript
const eventSource = new EventSource('/test/ai/stream');
eventSource.onmessage = (event) => {
    // event.data 就是去掉 data: 后的内容
    document.getElementById('output').innerHTML += event.data;
};
```

## 5. 编译问题

### 5.1 Java版本不支持文本块

**错误信息：**
```
java: -source 8 中不支持 文本块 (请使用 -source 15 或更高版本以启用 文本块)
```

**原因：**
- pom.xml中配置了Java 17，但IDE/编译器可能没识别
- 文本块（Text Blocks）是Java 15+的特性

**解决方案：**
添加maven-compiler-plugin明确指定Java版本：

```xml
<build>
    <plugins>
        <plugin>
            <groupId>org.apache.maven.plugins</groupId>
            <artifactId>maven-compiler-plugin</artifactId>
            <configuration>
                <source>17</source>
                <target>17</target>
                <encoding>UTF-8</encoding>
            </configuration>
        </plugin>
    </plugins>
</build>
```

### 5.2 TikaException未捕获

**错误信息：**
```
java: 未报告的异常错误org.apache.tika.exception.TikaException; 必须对其进行捕获或声明以便抛出
```

**原因：**
- `tika.parseToString()` 会抛出 `TikaException`
- 代码只捕获了 `IOException`

**解决方案：**
```java
// 修复前
catch (IOException e)

catch (IOException | org.apache.tika.exception.TikaException e)
```

### 5.3 VectorStatus找不到符号

**错误信息：**
```
java: 找不到符号
  符号:   变量 VectorStatus
  位置: 类 com.example.rag.entity.KnowledgeBaseEntity
```

**原因：**
- `VectorStatus` 是独立枚举类，不是 `KnowledgeBaseEntity` 的内部类
- 代码中使用了 `KnowledgeBaseEntity.VectorStatus.PENDING`

**解决方案：**
```java
// 错误
entity.setVectorStatus(KnowledgeBaseEntity.VectorStatus.PENDING);

// 正确
entity.setVectorStatus(VectorStatus.PENDING);
```

### 5.4 Document.setMetadata方法不存在

**错误信息：**
```
java: 找不到符号
  符号:   方法 setMetadata(java.util.Map<java.lang.String,java.lang.Object>)
  位置: 类 org.springframework.ai.document.Document
```

**原因：**
- Spring AI 1.0.0-M6的Document类没有setMetadata方法
- metadata需要在构造时传入

**解决方案：**
```java
// 错误
chunk.setMetadata(metadata);

// 正确 - 创建新Document
new Document(chunk.getText(), metadata);
```

## 6. 事务问题

### 6.1 事务回滚导致数据不一致

**错误信息：**
```
org.springframework.transaction.UnexpectedRollbackException: Transaction silently rolled back because it has been marked as rollback-only
```

**原因：**
- 多个@Transactional方法嵌套
- 内层事务失败标记为rollback-only
- 外层事务捕获异常后继续保存，但事务已被标记回滚

**解决方案：**
移除所有@Transactional注解，让方法在无事务环境下运行：

```java
// 移除@Transactional
public KnowledgeBaseEntity reprocessWithContent(Long id, String content) {
    // ...
}
```

---

### 6.2 事务设计模式选择

**问题背景：**
- `knowledge_bases` 表由 JPA 管理
- `vector_store` 表由 Spring AI 的 VectorStore 管理
- 两个表的事务管理方式不同

**两种设计模式：**

| 模式 | 说明 | 适用场景 |
|------|------|----------|
| 强一致性 | 向量化失败则回滚knowledge_bases插入 | 单体应用，同一数据库 |
| 最终一致性 | 向量化失败保留记录，标记为FAILED | 分布式系统，不同数据源 |

**本系统采用：最终一致性**

```java
@Transactional
public KnowledgeBaseEntity upload(...) {
    // 1. 保存记录（状态PENDING）← 事务内
    entity = knowledgeBaseRepository.save(entity);
    
    // 2. 向量化 ← VectorStore有自己的事务
    try {
        vectorService.vectorizeAndStore(entity.getId(), content);
        entity.setVectorStatus(VectorStatus.COMPLETED);
    } catch (Exception e) {
        entity.setVectorStatus(VectorStatus.FAILED);  // 失败标记，支持重试
    }
    
    // 3. 更新状态 ← 事务内
    return knowledgeBaseRepository.save(entity);
}
```

**为什么不用强一致性：**
- VectorStore内部有自己的事务管理
- 我们的事务和VectorStore事务嵌套会导致冲突
- Spring AI没有提供分布式事务支持

**最终一致性的优点：**
- 向量化失败时，记录保留（状态为FAILED）
- 用户可以通过"重新处理"按钮重试
- 更适合分布式系统

## 7. 数据库问题

### 7.1 vector_store表不存在

**错误信息：**
```
bad SQL grammar [DELETE FROM vector_store WHERE metadata->>'kb_id' = ?]
```

**原因：**
- Spring AI的pgvector没有自动创建表
- 需要配置 `initialize-schema: true`

**解决方案：**
```yaml
spring:
  ai:
    vectorstore:
      pgvector:
        initialize-schema: true
        dimensions: 1024
```

---

### 7.2 重复文件上传报错

**错误信息：**
```
ERROR: duplicate key value violates unique constraint "uk_xxx"
Key (file_hash)=xxx already exists.
```

**原因：**
- file_hash字段有唯一约束
- 重复上传同一文件会冲突

**解决方案：**
上传前检查hash，已存在则返回已有记录（幂等）：

```java
Optional<KnowledgeBaseEntity> existing = knowledgeBaseRepository.findByFileHash(fileHash);
if (existing.isPresent()) {
    return existing.get();
}
```

---

### 7.3 SQL中JSON操作符问题

**错误信息：**
```
bad SQL grammar [DELETE FROM vector_store WHERE metadata->>'kb_id' = ?]
```

**原因：**
- vector_store表不存在时，任何SQL都会失败
- 需要先检查表是否存在

**解决方案：**
```java
public int deleteByKnowledgeBaseId(Long knowledgeBaseId) {
    if (!tableExists()) {
        log.info("vector_store表不存在，跳过删除");
        return 0;
    }
    String sql = "DELETE FROM vector_store WHERE metadata->>'kb_id' = ?";
    return jdbcTemplate.update(sql, knowledgeBaseId.toString());
}
```
